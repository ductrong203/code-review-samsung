<note>**[Performance]** `GetFreePort` opens a listener to verify port availability. 

In `NewServerConfigs`, you are calling `GetFreePort` three times sequentially. While this is acceptable for small test suites, it is a common source of "flaky" tests in parallelized environments (race conditions where a port is grabbed by another process between the time `GetFreePort` closes its listener and the actual server starts). 

While this PR improves maintainability, consider documenting that this helper does not guarantee the port will remain free until the server starts, or consider if the server implementation can accept a listener object directly instead of a port number to avoid this race condition entirely.</note>
