<note>**[Performance]** The `communicate_with_quorum` function uses `FuturesUnordered` to process validator responses. While this is efficient for I/O, the logic inside the loop performs repeated `HashMap` lookups and `Vec` allocations for every incoming response. In a high-throughput scenario with many validators, this can lead to excessive memory pressure and CPU overhead.

**[Maintainability and Readability]** The label `'vote_wait` is used for a loop, but the logic inside the loop is becoming complex with multiple exit points and conditional breaks, which can hinder readability.

The grace period calculation `start_time.elapsed().mul_f64(grace_period)` is performed every time a quorum is reached. While not a critical bottleneck, it involves floating-point arithmetic inside a hot loop that could be simplified or moved outside the loop if the grace period is constant for the duration of the request.</note>
