<note>**[Code Defect]** The CurrentMetrics update logic is flawed because it is called at the end of the run() method, but the run() method can return early (line 49) or schedule tasks and exit without reaching the update logic. This leads to inconsistent metric states if the disk status changes during these early exits.

**[Performance]** The metrics are updated on every iteration of the disk check loop, even if the disk status has not changed. This results in redundant atomic operations (CurrentMetrics::add) which can cause unnecessary cache contention in high-concurrency environments.

The metric update logic is executed even when the disk check is in the 'retry' phase (lines 73-77). If the disk is temporarily inaccessible but not yet marked 'broken', the metrics will be updated based on the current state, potentially causing unnecessary metric churn or incorrect reporting if the state hasn't actually transitioned.

**[Maintainability and Readability]** The metrics update logic is duplicated at the end of the 'run' method. If more status flags are added in the future, this will become harder to maintain.</note>
