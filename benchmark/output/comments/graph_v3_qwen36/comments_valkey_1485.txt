<note>**[Performance]** The multiple blocks write path in `writeToReplica` allocates a stack array `struct iovec iov_arr[IOV_MAX]`. `IOV_MAX` is typically 16 or 64, but can be larger on some systems. Allocating large arrays on the stack in a hot path (replica writes) risks stack overflow under high concurrency or deep call stacks. Additionally, this array is allocated on every call to `writeToReplica` for multi-block writes, adding allocation overhead compared to a static or heap buffer if the size is bounded.

In the multiple blocks write path, `struct iovec iov_arr[IOV_MAX]` is allocated on the stack. If `IOV_MAX` is large, this consumes significant stack space. In a multi-threaded environment (IO threads), each thread has its own stack, so this is not a shared memory leak, but it increases memory pressure per thread. If `IOV_MAX` is not bounded or is very large, this could lead to stack exhaustion.

The code uses `struct iovec iov_arr[IOV_MAX]` which is a variable-length array (VLA) if `IOV_MAX` is not a compile-time constant. VLAs can have unpredictable stack usage and are not supported in all C standards. Even if `IOV_MAX` is a constant, large stack allocations in hot paths are risky.</note>

<note>In `unlinkClient`, `waitForClientIO(c)` is called before unlinking. This is a **Denial of Service (DoS)** vulnerability. If an IO thread is stuck or slow, or if the client has a massive amount of pending data, `waitForClientIO` will block the main thread indefinitely. This prevents the server from processing any other commands, effectively hanging the entire server instance. An attacker can trigger this by creating a client with a large amount of pending write data, causing the main thread to block on `waitForClientIO` during client unlinking (which happens on disconnect or timeout).</note>
