<note>The function `start_requirements_server` in `pip_install.rs` mounts two mocks for the same path `/requirements.txt`. The first handles authenticated requests (200), and the second handles unauthenticated requests (401). However, the second mock is mounted *after* the first. In `wiremock`, the order of matching is not guaranteed to be strict LIFO/FIFO in a way that ensures the specific matcher (basic_auth) is checked before the generic one. If the generic mock matches first, the test logic for 'Should fail without credentials' will fail because it expects a 401 but might get a 200 if the generic mock matches, or vice versa. More critically, the second mock `Mock::given(method("GET")).and(path("/requirements.txt"))` is a superset of the first. This creates a fragile test setup where the order of mock registration matters significantly and is not explicitly enforced by the library's public API guarantees for priority. It's better to use a single mock with a conditional response or ensure strict ordering.</note>
