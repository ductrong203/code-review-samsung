<note>**[Code Defect]** The `isWithinRoot` function in `fileUtils.ts` was changed to use `path.resolve` instead of `path.normalize`. `path.resolve` resolves relative paths against the current working directory, which can lead to incorrect security checks if the input path is relative or if the working directory changes. The original `normalize` approach was safer for absolute path validation. This change breaks the security boundary check.

**[Security Vulnerability]** The `isWithinRoot` function now uses `path.resolve()` on the input `pathToCheck`. This is a security regression because `path.resolve()` normalizes the path (e.g., resolving `..`), which can allow an attacker to bypass the root directory check by providing a path like `/root/../../etc/passwd` which resolves to `/etc/passwd`. The previous implementation used `path.normalize()` which is safer for this specific containment check as it doesn't resolve parent directory traversals in the same way on all platforms, or at least the previous logic was more explicit about the containment check. By resolving the path first, the check `startsWith` might fail if the resolved path is outside the root, but the tool might still operate on the resolved path if the validation is bypassed or if the resolved path is used elsewhere. More critically, `path.resolve` makes relative paths absolute relative to CWD, which might not be the intended root, leading to unintended file access.

**[Performance]** The `isWithinRoot` function now uses `path.resolve` on both the path and root directory. `path.resolve` performs expensive string manipulation and filesystem checks (like resolving `..` and `.`) on every call. Since the root directory is typically an absolute path already, and the check path is often absolute, this adds unnecessary overhead compared to `path.normalize` which is lighter. This function is called in validation paths for multiple tools (edit, ls, read-file, write-file, glob, grep), making it a hot path.</note>
