<note>The code uses `snprintf` with `%s` format specifier on `c->querybuf + c->qb_pos`. While the buffer `buf` is fixed-size (256 bytes), `snprintf` truncates output, preventing a buffer overflow. However, if the query buffer contains binary data or null bytes, `snprintf` with `%s` will stop at the first null byte, potentially logging incomplete data. More critically, if `c->querybuf` is not null-terminated (which `sds` strings are, but relying on pointer arithmetic `c->querybuf + c->qb_pos` assumes validity), it could lead to reading out of bounds if `qb_pos` is invalid. The primary risk here is information disclosure if the redaction logic fails or is bypassed, but the current change actually mitigates this by adding redaction. The remaining non-redacted path still exposes raw user input in logs, which is a privacy/compliance issue (A09 Logging Failures).</note>

Previously, this loop was always executed. While this change reduces work when redaction is enabled, it introduces a branch misprediction penalty in the common case where data is logged. More importantly, if `hide_user_data_from_log` is false, the loop runs on the potentially large `buf` array. If `buf` is large, this is an O(N) operation on every protocol error. Consider if this transformation is necessary for every error or if it can be deferred or optimized.</note>
