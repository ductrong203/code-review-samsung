<note>**[Performance]** String concatenation in a loop using `+=` is inefficient for large streams. In a streaming context where `textResponse` can grow significantly, this creates many intermediate string objects. Use a `StringBuilder` or array join pattern for better performance.

**[Security Vulnerability]** The code accumulates text responses from the model into `textResponse` and streams it via `onMessage`. If the model generates a very long response, this string accumulation could lead to high memory usage (DoS). Additionally, if the text contains sensitive information, it is exposed to the client. There is no rate limiting or size cap on the streamed text chunks.</note>

<note>**[Security Vulnerability]** The code serializes tool arguments using JSON.stringify and includes them in streamed messages via onMessage. While there is a length truncation (250 chars), this exposes internal tool execution details and arguments to the client. If tool arguments contain sensitive data (PII, tokens, secrets), this information is leaked through the streaming channel. This is an information disclosure vulnerability.

**[Maintainability and Readability]** The `onMessage` callback is invoked with a formatted string that includes newlines and markdown-like formatting (e.g., `**Executing tool...**`). This formatting logic is embedded directly in the streaming handler. If the consumer of `onMessage` expects raw text or a different format, this will break. It's better to pass structured data (tool name, args) and let the consumer format it, or provide a separate callback for structured events.</note>
