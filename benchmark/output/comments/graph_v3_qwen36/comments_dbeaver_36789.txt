The SQL query uses string concatenation to build the query string, but the parameters are correctly passed using setString. However, the query construction itself is complex and spans multiple lines, increasing the risk of future injection if someone modifies the concatenation logic. The current implementation is safe because it uses parameterized queries for the actual values (table.getName() and table.getContainer().getName()), but the multi-line string concatenation style is error-prone.

**[Performance]** The SQL query string is constructed using string concatenation with newlines inside the method `prepareObjectsStatement`. This method is called every time the partition cache is refreshed. While the concatenation itself is minor, constructing the SQL string repeatedly is less efficient than defining it as a static final constant. Additionally, the query selects `tp.*` which retrieves all columns from `TABLE_PARTITIONS`, potentially including unused data, increasing network and memory overhead.</note>
