<note>**[Performance]** The cleanup loop for submitted command buffers is now nested inside the 'presenting' check. If the application runs in a headless mode or performs compute-only work without presenting, the submitted command buffers will never be cleaned up. This leads to a memory leak of command buffers and fences, and the GPU queue will eventually stall because the submission queue grows unbounded.

**[Security Vulnerability]** In VULKAN_Submit, cleanup operations (command buffer cleanup, empty allocation deallocation, pending destroys) are now guarded by 'if (presenting)'. In headless or non-presenting contexts, these resources are never cleaned up. This leads to a memory leak and eventual resource exhaustion (OOM), which can be triggered by an attacker or heavy workload to cause a Denial of Service.</note>
