<note>**[Performance]** The refactored logic in `communicate_with_quorum` introduces a potential performance regression. The original code waited for the grace period if a quorum was reached OR if it was clear no key could reach quorum. The new code breaks immediately if no key can reach quorum (`highest_key_score + remaining_votes < committee.quorum_threshold()`). This means that in scenarios where no quorum is possible, the function returns immediately without waiting for the grace period. While this might seem faster, it changes the behavior of the quorum communication protocol, potentially leading to inconsistent state or missed error information that was previously collected during the grace period. More critically, if the intention was to optimize for the 'no quorum' case, the logic `highest_key_score + remaining_votes < committee.quorum_threshold()` is computed in every iteration of the loop. If `remaining_votes` is not updated efficiently or if `highest_key_score` is expensive to compute, this could be a bottleneck. However, the primary issue is the behavioral change which might be a bug, but from a perf perspective, if the 'no quorum' case is common, this is an optimization. But if the 'no quorum' case is rare, the overhead of the check is negligible. The real issue is that the original code waited for the grace period in BOTH cases (quorum reached OR no quorum possible). The new code only waits if quorum is reached. This means errors are reported faster, which is good. But if the grace period was intended to collect more error info for debugging, this is a loss. Let's look closer. The original code: `if end_time.is_none() && (highest_key_score >= committee.quorum_threshold() || highest_key_score + remaining_votes < committee.quorum_threshold())`. This means if EITHER condition is true, it sets `end_time`. The new code: `if highest_key_score + remaining_votes < committee.quorum_threshold() { break }`. Then `if end_time.is_none() && highest_key_score >= committee.quorum_threshold() { set end_time }`. So if no quorum is possible, it breaks immediately. If quorum is reached, it waits. This is a significant change in behavior. Is it a performance issue? If the 'no quorum' case is common, this is a HUGE performance win. If it's rare, it's a behavioral bug. Given the PR description says 'improve the handling of unreachable quorum', this is likely an intentional optimization. So maybe not a perf issue. Let's look for other issues.

**[Maintainability and Readability]** The refactored logic in `communicate_with_quorum` introduces a potential regression in behavior. The original code set `end_time` if a key reached quorum OR if it became clear no key could reach quorum. The new code breaks early if no key can reach quorum, but only sets `end_time` if a key reaches quorum. The logic is now asymmetric and potentially much slower in failure cases.</note>

<note>Logic error in grace period calculation: `start_time.elapsed()` is called inside the loop. As the loop runs, `elapsed()` grows, so the grace period deadline (`end_time`) keeps extending further into the future on every iteration. This causes the loop to wait significantly longer than intended (potentially infinite if the loop runs long enough), instead of waiting a fixed grace period relative to when the quorum was first reached. Fix: calculate the grace period deadline once when the quorum is reached, outside the loop.</note>
