<note>**[Maintainability and Readability]** The `getDates` method is used to populate motion asset creation fields. However, the method relies on `firstDateTime` which may return `undefined` if no valid date is found in EXIF tags. The code does not handle this case, leading to `undefined` values being assigned to `fileCreatedAt`, `fileModifiedAt`, and `localDateTime`. This is a regression from the previous logic which explicitly fell back to `asset.fileCreatedAt`. This makes the code fragile and potentially introduces null/undefined values into the database.

**[Performance]** In the `applyMotionPhotos` method within `metadata.service.ts`, `this.getDates` is called for every motion asset creation. This function performs date parsing and validation. If this path is triggered frequently (e.g., during bulk processing of videos with motion photos), it adds unnecessary overhead compared to the previous direct assignment. Consider if `getDates` can be optimized or if the result can be cached if multiple assets share similar metadata.</note>
