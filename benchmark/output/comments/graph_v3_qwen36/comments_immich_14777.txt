<note>**[Security Vulnerability]** The change `deleteOnDisk: !asset.isOffline` introduces a logic flaw where assets that are NOT offline (i.e., online assets) will have `deleteOnDisk` set to `false`. This means when an online asset is deleted, its file is NOT removed from disk, leading to data accumulation and potential disk exhaustion (DoS). Conversely, if `isOffline` is true, it deletes the file, which might be intended, but the primary risk is the failure to delete online assets.

**[Performance]** The change to set deleteOnDisk based on asset.isOffline introduces a potential performance issue if the asset object is not fully hydrated with the isOffline flag before this check. If isOffline is null or undefined, the logic might default incorrectly or require an additional database query to resolve the status, leading to N+1 query patterns or unexpected behavior in high-throughput deletion jobs.

While the PR description claims to fix data loss by NOT deleting offline files, the implementation `!asset.isOffline` is ambiguous and likely incorrect for the 'online' case. If an asset is online, `isOffline` is false, so `deleteOnDisk` becomes false. This means normal user deletions will leave orphaned files on disk, which is a resource exhaustion vulnerability.</note>
