<note>**[Performance]** The code performs set union operations (using the | operator) on SUPPORTED_DATASET_PATHS inside the validation logic. Since this validation runs every time the benchmark starts, creating new set objects repeatedly is inefficient. These sets should be pre-computed as constants or class-level attributes to avoid redundant allocations.

**[Maintainability and Readability]** The validation logic in validate_args uses hardcoded set operations to check backend compatibility. This violates the Open/Closed principle; every time a new dataset is added, this function must be modified. This logic should be encapsulated within the dataset classes themselves (e.g., a 'required_backend' property).</note>
